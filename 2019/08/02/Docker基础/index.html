Docker简介

基础概念

Docker是一个用户开发、发布和运行应用的开放平台，Docker使您能够将应用程序与基础架构分离以便快速交付应用。使用Docker，您可以和管理应用程序一样管理基础架构。通过利用Docker的方法快速地交付、测试和部署代码，您可以显著地减少编写代码和在生产环境中运行代码的延迟。

我可以使用Docker来做什么？

快速、一致地交付您的应用

由于Docker允许开发人员使用本地容器在标准化环境中工作，这些容器提供应用程序和服务，从而简化了开发生命周期。

响应式部署和扩展

Docker的基于容器的平台允许高度可移植的工作负载，Docker容器可以运行在开发人员的本地电脑、数据中心中的物理或虚拟机、云提供商或混合环境中

在相同的硬件上运行更多的工作负载

Docker轻巧而快速。它为基于管理程序的虚拟机提供了一种可行且经济高效的替代方案，因此您可以使用更多的计算容量来实现业务目标。Docker非常适合高密度环境以及需要用更少资源完成更多工作的中小型部署。

Docker架构

Docker使用客户端-服务器架构，其中

  • 服务器是一种长时间运行的程序，称为守护进程(dockerd命令)
  • 一个REST API，它指定了一些接口，程序可以使用这些接口与守护进程对话并指示它做什么。
  • 命令行接口(CLI)客户端(docker命令)。

镜像(Image)

基础概念

一个镜像是只读模块，包含运行应用程序所需的所有内容——代码，运行环境，库，环境变量，和配置文件。通常，一个镜像以另一个镜像为基础，并添加一些额外的自定义内容。要创建自己的镜像，你可以通过一些简单的语法来创建Dockerfile，其中定义了创建和运行镜像的步骤。换句话说，镜像只读图层（layer）组成，其中每个图层代表一个Dockerfile的指令。这些图层是堆叠的并且每个图层是前一层变化的增量。

容器(Container)

基础概念

  • 容器是镜像的运行时实例——镜像在被执行时在内存中变为什么（即具有状态的镜像或用户进程）。您可以使用命令docker ps查看正在运行中的容器列表，就像在Linux中一样。默认情况下，容器与其他容器、它的主机间相对隔离。您可以控制容器与网络、存储或其他基础子系统与其他容器或主机的隔离程度
  • 当运行镜像生成容器时，在底部图层的顶部添加了一个新的可写图层（container layer）,对正在运行的容器作出的所有更改，比如写入新文件，修改已存在的文件，删除文件都将被写入此可写容器层。

仓库(Repository)

基础概念

注册表是仓库的集合，仓库是镜像的集合——类似于Github存储库，除了代码已经构建。注册表上的账户可以创建多个仓库。docker cli默认使用Docker的公共注册表。

DockerFile

基础概念

DockerFile定义了容器内的环境中发生了什么。对网络接口和磁盘驱动器等资源的访问在此环境中进行虚拟化，该环境与系统的其他部分隔离，所以您需要将外部的世界映射到端口上，并且明确您想“复制”到该环境中的文件。

使用.dockerignore进行排除

要排除与构建无关的文件（不重构源存储库），请使用.dockerignore文件。此文件支持与.gitignore文件类似的排除模式。有关创建一个.dockerignore的信息，请参阅.dockerignore。

使用多级构建

多级构建允许您大幅减小最终镜像的大小，而无需减少中间层和文件的数量。
例如，如果您的构建层包含多个图层，则可以从较不频繁更改（以确保构建缓存可重用）到更频繁更改的顺序对它们进行排序：

  • 安装构建应用程序所需要的工具
  • 安装或更新库依赖项
  • 生成您的应用程序

解藕应用程序

每个容器应当只有一个关注点，将应用程序分离到多个容器中，使其更容易水平扩展和重用容器。例如，一个web应用程序可能包含三个独立的容器，每个容器都有其独特的镜像，以分离的方式管理web应用程序、数据句和内容缓存。

最小化层数

在旧版本的Docker中，最大限度地减少镜像中的图层数量以确保它们具有高性能非常重要。添加了以下功能以减少此限制：

  • 只有RUN，COPY，ADD指令会创建图层，其他指令会创建临时图层，不会增加最终构建的大小。
  • 在可能的情况下，使用多级构建，并且只将您需要的工件复制到镜像中。这允许您在中间构建阶段中包含工具和调试信息，而不会增加最终图像的大小。

利用构建缓存

当构建镜像时，Docker会逐步遍历Dockerfile中的指令，按定义顺序执行每条指令。Docker会在其缓存中查找可以重用的现有映像，而不是创建新的（重复）映像。
如果您根本不想使用缓存，您可以在docker build命令上使用--no-cache=true选项。但是，如果您想让Docker使用其缓存，重要的是要理解何时可以找到匹配的图像，何时不能找到匹配的图像。Docker遵循的基本规则如下:

  • 从已存在缓存中的父镜像开始，下一条指令会从所有从该基础镜像中衍生的子镜像进行比较，以查看它们中的一个是否完全使用相同的指令构建。如果不是，则缓存无效。
  • 在大多数情况下，简单的将Dockerfile与子镜像中的指令进行比较就足够了。但是，某些说明需要更多的检查和解释。
  • 对于ADD和COPY指令，检查图像中文件的内容，并计算每个文件的校验和。在这些校验和中不考虑最后修改时间和最后访问时间。在缓存找寻期间，将校验和与现有镜像中校验和进行比较。如果文件中的内容（例如内容和元数据）发生任何改变，缓存将失效。
  • 除了ADD和COPY命令以外，除了添加和复制命令外，缓存检查不会查看容器中的文件来确定缓存匹配。例如，在处理RUN apt-get -y update命令时，不会检查容器中更新的文件，以确定缓存是否命中。在这种情况下，仅使用命令字符串本身来查找匹配项。

CMD

CMD指令用于运行镜像中包含的软件以及任何参数。CMD几乎总是以CMD ["executable", "param1", "param2"…]的形式被使用。但是，如果该镜像是用于服务的，比如Apache和Rails，您应当运行类似CMD ["apache2","-DFOREGROUND"]。实际上，建议将这种形式的指令用于任何基于服务的镜像。
在大多数其他情况下，CMD应当给出一个交互型的shell，比如bash，python或perl。比如，CMD ["perl", "-de0"], CMD ["python"], or CMD ["php", "-a"]。用这种形式意味着，当您执行像docker run - python这样的东西时，您将被放入一个可用的shell中，准备就绪。

CMD有三种形式：

  • CMD ["executable","param1","param2"] （执行形式，这是首选形式）

    CMD的主要目的是为执行容器提供默认默认值。这些默认值可以包含可执行文件，他们也可以忽略可执行文件，在这种情况下，您必须指定一条ENTRYPOINT指令。

ONBUILD

ONBUILD命令在当前Dockerfile构建完成后执行。ONBUILD指令在任何从当前镜像中导出的镜像中执行。可以将ONBUILD命令视为父Dockerfile给子Dockerfile的指令。
Docker构建在子Dockerfile中的任何命令之前执行ONBUILD命令。
任何指令都可以注册成为一个触发器。
如果您要构建一个镜像并将其作为构建其他镜像的基础，这非常有用，例如一个应用程序构建环境或一个守护进程，它可以使用特定于用户的配置进行定制。
例如，如果您的镜像是一个可重用的Python应用程序构建器，则需要将应用程序源代码添加到特定的目录，并且可能在这之后调用构建脚本。您现在不能只是调用ADD和RUN，因为您现在还没有访问应用程序的源代码，每个应用程序构建这都将是不同的。您可以简单地为应用程序开发人员提供一个Dockerfile样例文件来复制粘贴到他们的应用程序中，但是这是低效的，容易出错的，而且很难更新，因为它与应用程序特定的代码混合在一起。
解决方案是使用ONBUILD来注册要在下一个构建阶段运行的预先指令。
以下是他的工作原理：

  1. 当遇到ONBUILD指令时，构建器会向正在构建的镜像元数据添加一个触发器。该指令不会影响当前构建。
  2. 在构建结束后，所有触发器的列表存储在镜像清单中的OnBuild键下,可以使用docker inspect命令检查。
  3. 之后可以使用FROM指令来将该镜像作为一个新的构建的基础。作为处理FROM指令的一部分，下游构建器将查找ONBUILD触发器，并且按照他们注册的顺序执行他们。如果任何触发器失败，则会导致FROM终止，反过来导致构建失败。如果所有的构建器成功，FROM指令完成并且该构建继续照常进行。
  4. 触发器在最终执行完成后从镜像中移除。
